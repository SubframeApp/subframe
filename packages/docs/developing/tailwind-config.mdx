Subframe integrates with Tailwind CSS by extending your configuration with design tokens from your Subframe theme. When you run `npx @subframe/cli init`, the CLI automatically configures Tailwind to use your Subframe theme colors, typography, spacing, and other design tokens.

## How Subframe extends Tailwind

Subframe doesn't replace your Tailwind configuration—it extends it. The CLI adds Subframe's design tokens to your `tailwind.config` file while preserving any existing configuration you have.

Here's what Subframe adds:

- **Colors** - Your theme's color palette (brand, neutral, error, warning, success, etc.)
- **Typography** - Font families, sizes, and weights from your theme
- **Spacing** - Custom spacing scale that matches your design system
- **Border radius** - Consistent border radius values
- **Box shadows** - Shadow tokens for elevation
- **Breakpoints** - Mobile breakpoint (767px and below)
- **Content paths** - Ensures Tailwind scans Subframe component files

All of this happens automatically during installation.

## Tailwind config location

The CLI modifies your existing `tailwind.config.js` or `tailwind.config.ts` file at the root of your project. If you don't have one, the CLI creates it.

After installation, your config looks something like this:

```ts tailwind.config.ts
import type { Config } from "tailwindcss"

const config: Config = {
  content: [
    "./src/**/*.{js,ts,jsx,tsx,mdx}",
    // Subframe adds this line:
    "./node_modules/@subframe/core/**/*.{js,ts,jsx,tsx}",
  ],
  theme: {
    extend: {
      colors: {
        brand: {
          50: "#f0f9ff",
          100: "#e0f2fe",
          // ... rest of brand colors
        },
        neutral: {
          // ... neutral colors
        },
        // ... other color tokens
      },
      fontFamily: {
        sans: ["Inter", "sans-serif"],
        // ... other font families
      },
      fontSize: {
        // ... font sizes from your theme
      },
      spacing: {
        // ... spacing scale
      },
      // ... other theme extensions
    },
  },
  plugins: [
    require("@tailwindcss/container-queries"),
  ],
}

export default config
```

## Theme tokens in Tailwind

All design tokens from your Subframe theme are available as Tailwind classes.

### Colors

Your theme colors become Tailwind color utilities:

```tsx
<div className="bg-brand-100 text-brand-900">
  Brand colors
</div>

<div className="bg-neutral-50 border border-neutral-200">
  Neutral grays
</div>

<div className="bg-error-500 text-white">
  Error state
</div>
```

Use the full color scale from 50 (lightest) to 950 (darkest), just like Tailwind's default colors.

### Typography

Font families, sizes, and weights from your theme:

```tsx
<h1 className="font-sans text-4xl font-bold">
  Heading with theme typography
</h1>

<p className="text-base font-normal">
  Body text using theme font
</p>
```

### Spacing

Your theme's spacing scale:

```tsx
<div className="p-6 gap-4 m-8">
  Padding, gap, and margin using theme spacing
</div>
```

Subframe typically uses a spacing scale like: 4, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80, 96 in pixels.

### Borders

Border radius values:

```tsx
<div className="rounded-md border border-neutral-200">
  Rounded corners using theme tokens
</div>
```

### Shadows

Box shadow tokens for elevation:

```tsx
<div className="shadow-sm">
  Subtle shadow
</div>

<div className="shadow-lg">
  Large shadow
</div>
```

## Custom Tailwind configuration

You can add your own Tailwind configuration alongside Subframe's tokens.

### Extending the config

Add custom values to the `extend` object:

```ts tailwind.config.ts
const config: Config = {
  theme: {
    extend: {
      // Subframe's tokens are here...

      // Add your custom values:
      colors: {
        custom: {
          purple: "#8b5cf6",
          orange: "#f97316",
        },
      },
      spacing: {
        '128': '32rem',
        '144': '36rem',
      },
      animation: {
        'fade-in': 'fadeIn 0.5s ease-in-out',
      },
      keyframes: {
        fadeIn: {
          '0%': { opacity: '0' },
          '100%': { opacity: '1' },
        },
      },
    },
  },
}
```

Your custom configuration works alongside Subframe's tokens without conflicts.

### Adding plugins

Add Tailwind plugins as needed:

```ts tailwind.config.ts
const config: Config = {
  plugins: [
    require("@tailwindcss/container-queries"), // Subframe requires this
    require("@tailwindcss/forms"),              // Add your plugins
    require("@tailwindcss/typography"),
  ],
}
```

<Warning>
Keep `@tailwindcss/container-queries` in your plugins array. Subframe components rely on container queries for responsive behavior.
</Warning>

## Content paths

The `content` array tells Tailwind which files to scan for class names. Subframe adds its own path so Tailwind includes classes used by Subframe components.

Make sure both paths are present:

```ts tailwind.config.ts
const config: Config = {
  content: [
    "./src/**/*.{js,ts,jsx,tsx,mdx}",              // Your files
    "./node_modules/@subframe/core/**/*.js",        // Subframe components
  ],
}
```

Add more paths if you have components elsewhere:

```ts
content: [
  "./src/**/*.{js,ts,jsx,tsx,mdx}",
  "./app/**/*.{js,ts,jsx,tsx,mdx}",               // Next.js app directory
  "./components/**/*.{js,ts,jsx,tsx}",            // Components folder
  "./node_modules/@subframe/core/**/*.js",
],
```

## Responsive design

Subframe adds a `mobile` breakpoint for responsive design.

### Mobile breakpoint

Subframe's mobile breakpoint activates at **767px and below**:

```tsx
<div className="flex flex-row mobile:flex-col gap-6 mobile:gap-4">
  {/* Horizontal on desktop, vertical on mobile */}
  {/* Gap of 24px on desktop, 16px on mobile */}
</div>
```

The `mobile:` prefix applies styles to screens 767px and below. This matches how you set mobile overrides in the Subframe editor.

### Tailwind's default breakpoints

Tailwind's default breakpoints still work:

```tsx
<div className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3">
  {/* 1 column on mobile, 2 on tablet, 3 on desktop */}
</div>
```

Use whichever breakpoint system fits your needs. Many projects use both—`mobile:` for simple mobile overrides and Tailwind's default breakpoints for more granular responsive design.

## Syncing config updates

When you update your theme in Subframe and run `npx @subframe/cli sync`, the CLI updates your Tailwind config with new token values.

### What gets updated

- **Color values** - If you change brand colors in Subframe
- **Typography** - If you update fonts or sizes
- **Spacing** - If you modify the spacing scale
- **Other tokens** - Any theme changes sync to Tailwind

### Custom configuration

The CLI tries to preserve your custom configuration when syncing. However, if you've heavily modified the sections Subframe manages, conflicts can occur.

### Preventing overwrites

If you need to prevent Subframe from updating specific parts of your config, you have a few options:

1. **Keep custom config separate** - Add custom values in separate keys that Subframe doesn't touch
2. **Use CSS variables** - Define custom values as CSS variables instead of Tailwind config
3. **Lock the file** - In rare cases, you can prevent syncing the config (but you'll miss theme updates)

## Dark mode setup

Subframe supports dark mode through CSS variables. Your Tailwind config doesn't need changes for dark mode—it works through the theme system.

For complete dark mode setup, see [Dark Mode](/theme/dark-mode).

## Tailwind v4 support

Subframe supports both Tailwind CSS v3 and v4. The CLI detects your Tailwind version and generates the appropriate configuration.

**Tailwind v3** uses `tailwind.config.js` or `tailwind.config.ts` files.

**Tailwind v4** uses CSS-based configuration with `@theme` directives. The CLI generates theme tokens as CSS variables that work with v4's new system.

You don't need to do anything special—the CLI handles version differences automatically.

## Best practices

### Don't delete Subframe's paths

Keep the Subframe content path in your config:

```ts
"./node_modules/@subframe/core/**/*.js"
```

Without this, Tailwind won't include classes used by Subframe components, causing missing styles.

### Use `extend` not `theme`

Always add custom configuration inside `theme.extend`, not directly under `theme`. This prevents overwriting Subframe's tokens:

```ts
// ✓ Good - extends Subframe's config
theme: {
  extend: {
    colors: {
      custom: { ... }
    }
  }
}

// ✗ Bad - overwrites everything
theme: {
  colors: {
    custom: { ... }
  }
}
```

### Keep config in version control

Commit your `tailwind.config` file to git so your team shares the same configuration.

### Test after theme updates

After syncing theme changes from Subframe, check that:
- Colors still look correct
- Typography renders properly
- Spacing hasn't broken layouts
- Custom configuration still works

## Troubleshooting

### Styles not applying

If Tailwind classes don't work:

1. Check that your dev server is running
2. Verify the content paths include your files
3. Make sure you imported the Tailwind CSS file in your app
4. Clear your build cache and restart the dev server

### Config overwritten by sync

If syncing overwrites your custom configuration:

1. Move custom config to separate keys that Subframe doesn't use
2. Keep a backup of your custom configuration
3. Re-apply custom config after syncing
4. Consider using CSS variables for custom values

### Build errors

If you see Tailwind build errors:

1. Check for syntax errors in `tailwind.config`
2. Verify all required plugins are installed: `npm install @tailwindcss/container-queries`
3. Make sure content paths use forward slashes (even on Windows)
4. Clear `node_modules` and reinstall if corrupted

### Classes not purged correctly

If production builds include unused classes:

1. Verify content paths cover all files using Tailwind
2. Don't use string concatenation for class names: `"bg-" + color` won't work
3. Use full class names: `"bg-brand-500"` not dynamic strings
4. Check safelist if you need to preserve specific classes

### Mobile breakpoint not working

If `mobile:` classes don't apply:

1. Verify the mobile breakpoint exists in your config
2. Check that the breakpoint is defined correctly (max-width: 767px)
3. Test in browser dev tools with responsive mode
4. Make sure you're using `mobile:` prefix not `sm:` for Subframe's mobile breakpoint
